## What We're Building

A 5-phase passive OSINT (Open Source Intelligence) workflow integrated into the existing Django 5 + Next.js 14 app. The output is a professional `.docx` report a cybersecurity sales rep gifts to prospects to demonstrate Pellera Technologies' capabilities.

**What makes this hard:** The workflow has two mandatory human-in-the-loop gates:
1. The user must run terminal commands locally (dig, whois, crt.sh) and paste results back
2. The user must visit DNSDumpster/Shodan in a browser, take screenshots, and upload them

Everything else in the app is fire-and-forget async. OSINT is not.

## Architecture Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| App structure | New `osint` Django app | Separate lifecycle, models, state. Refs `research.ResearchJob` via nullable FK (same pattern as `ideation`) |
| LangGraph | Separate workflow with `MemorySaver` checkpointer | Interrupt/resume requires checkpointing. Existing research workflow has none. MemorySaver for dev (upgrade to PostgresSaver pre-prod) |
| DNS/WHOIS | Python libraries only (`dnspython`, `python-whois`, ARIN REST) | No subprocess calls — eliminates command injection. Architecturally cannot perform active scanning |
| Report | `python-docx` with Builder pattern | 10 sections, independent section classes, template-first approach |
| Shodan | User screenshots only (no server-side API) | No API key required; degrades gracefully |
| Frontend workflow | Phase stepper (not tabs) | Sequential human-gated phases communicate better with a stepper; tabs imply simultaneous availability |

## The 5 Phases

### Phase 1 — Web Research (automated)
Gemini grounded queries: company profile, breach history, job postings intel, vendor relationships, regulatory framework, leadership thought leadership. Reuses `run_parallel_grounded_queries()` from `research/services/grounding.py` with OSINT-focused prompts.

### Phase 2 — DNS & Infrastructure OSINT (semi-automated)
**Server does automatically:** crt.sh Certificate Transparency query, DNS records (MX, TXT, NS, A, SOA) via dnspython, DMARC check, SPF/DKIM/DMARC assessment, WHOIS, ARIN lookup for discovered IPs.

**User runs locally:** Reverse DNS sweeps on discovered subnets, SPF chain expansion, subdomain resolution checks, Wayback Machine queries. Server generates these commands; user copies, runs, pastes back output. Up to 3 rounds.

### Phase 3 — Browser-Based OSINT (user action required)
User visits DNSDumpster and Shodan, takes screenshots, uploads to app. Gemini vision analyzes each image and extracts structured data (IPs, hostnames, tech stack, ports).

### Phase 4 — Analysis & Enrichment (automated)
Gemini synthesis of all Phase 1–3 findings: infrastructure map, technology stack fingerprint, risk matrix (Likelihood × Impact), severity table, service mappings.

### Phase 5 — Final Report (triggered by user)
10-section `.docx` generated by `OsintReportBuilder`. User explicitly triggers this after reviewing findings.

## OsintJob Status State Machine

```
pending
  → phase1_research
  → phase1_complete
  → phase2_auto
  → awaiting_terminal_output   ← USER ACTION: run commands, paste output
  → phase2_processing
  → awaiting_screenshots       ← USER ACTION: upload DNSDumpster + Shodan screenshots
  → phase3_processing
  → phase4_analysis
  → phase5_report              ← USER TRIGGERS: "Generate Report" button
  → completed
  → failed (from any state)
```

## Report Structure (10 sections)

1. Cover Page — CONFIDENTIAL, Pellera branding, date
2. Executive Summary — financial risk context (IBM/Ponemon breach costs), risk heat map, severity table
3. Remediation Action Plan — 0-30 days, 30-90 days, 90-180 days with hours + cost benchmarks + ROSI
4. Strategic Security Roadmap — 12-month priorities + 3-year vision
5. Detailed Findings per Entity — org profile, attack surface, email security, tech stack, regulatory
6. Regulatory Compliance Landscape — applicable regulations mapped to findings
7. Appendix A: Pellera Engagement Proposal — findings → 9 service offerings
8. Appendix B: OSINT Methodology — all commands used with results
9. Appendix C: Infrastructure Maps — embedded screenshots

## Integration with Existing Research

When launched from an existing research job (research results page → 9th tab):
- `OsintJob.research_job` FK is set
- Phase 1 node reads `prior_research_context` from the linked `ResearchReport` — avoids redundant Gemini calls
- Creation form pre-fills: prospect name, primary domain, industry, linked research job ID

Standalone OSINT (no prior research) is also fully supported — Phase 1 runs the full research pass.

## Security Invariants

- **No subprocess calls.** All DNS/WHOIS uses Python libraries only.
- **Domain validation on every input.** Regex + length check + reject `.local`/`.internal`/`.corp`.
- **No active scanning capability.** `dnspython` performs standard queries only — no AXFR, no port scanning.
- **Upload validation.** ImageField: PNG/JPEG/WebP via Pillow verify, 10MB max, 20 files max.
- **Terminal output is never executed.** Treated as untrusted text, parsed by Gemini, stored as-is.
- **Rate limiting.** `osint_execute: 5/hour` on execute endpoint.
